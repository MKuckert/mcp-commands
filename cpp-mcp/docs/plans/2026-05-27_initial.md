# Project Plan: CLIBridge â€” Executable Directory Scanner

## đźŽŻ Objective

Extend `clibridge` so that on startup it recursively scans a user-supplied directory for
executable files and registers each one as an MCP tool. When an agent calls such a tool,
the server spawns the executable, optionally pipes `stdin` data into it, and returns
`stdout`, `stderr`, and the exit code. Non-zero exit codes and process signals are
surfaced as MCP errors (`isError: true`).

## đź›  Requirements & Decisions

- **Frameworks:** `cpp-mcp` core library (already in-tree), C++17 `<filesystem>`.
- **Chosen Libraries:** `reproc++` via CMake `FetchContent` for cross-platform process
  spawning with timeout support.
- **Error Handling Strategy:**
  - **Timeout (30 s constant):** Kill process, set `isError: true`, include partial
    `stdout`/`stderr` collected so far plus a timeout message.
  - **Executable deleted between scan and call:** `isError: true` with descriptive message;
    no re-scan triggered.
  - **Non-zero exit / signal:** `isError: true`; `stdout`, `stderr`, and `exitCode` still
    populated and returned so the caller can diagnose.
  - **`stdin` data not consumed by child:** No special handling; pipe closes naturally.
  - **No executables found in directory:** Server starts normally without tools; a warning
    is logged via `MCP_LOG_WARN` (or `std::cerr`).
  - **General reproc errors:** Map to `mcp::mcp_exception(internal_error, ...)` for
    unexpected OS-level failures.

- [ ] **Task 2: Implement `ExecutableScanner`**
  - **Description:** Create `clibridge/executable_scanner.h` (header-only for simplicity).
    Expose one function:
    ```cpp
    // Returns list of {tool_name, absolute_path} pairs.
    // tool_name = path relative to scan_root, e.g. "foo/bar/script.sh"
    std::vector<std::pair<std::string, std::filesystem::path>>
    scan_executables(const std::filesystem::path& scan_root);
    ```
    - Use `std::filesystem::recursive_directory_iterator`.
    - Include only regular files with the executable permission bit set
      (`(status.permissions() & fs::perms::owner_exec) != fs::perms::none`).
    - Derive `tool_name` as the lexically relative path string from `scan_root`.
    - If the directory does not exist, throw `std::runtime_error`.
    - If the result is empty, return empty vector (caller logs the warning).
  - **Review Criteria:** Unit test covers: normal tree with mixed files, empty dir,
    nested executables, non-executable files excluded, non-existent dir throws.

- [ ] **Task 3: Implement `ProcessRunner`**
  - **Description:** Create `clibridge/process_runner.h` (header-only). Expose:
    ```cpp
    struct RunResult {
        std::string stdout_data;
        std::string stderr_data;
        int         exit_code;   // meaningful only when !timed_out && !signal_killed
        bool        timed_out;
        bool        signal_killed;
    };

    RunResult run_process(
        const std::filesystem::path& executable,
        const std::string&           stdin_data,
        std::chrono::milliseconds    timeout = std::chrono::seconds(30));
    ```
    Implementation outline using `reproc++`:
    1. Build `reproc::options` with `redirect.in = REPROC_REDIRECT_PIPE`,
       `redirect.out = REPROC_REDIRECT_PIPE`, `redirect.err = REPROC_REDIRECT_PIPE`.
    2. Set `stop.first = { REPROC_STOP_TIMEOUT, 30000 }`,
       `stop.second = { REPROC_STOP_KILL, 5000 }` (deadline enforced by reproc drain loop).
    3. Write `stdin_data` then close stdin stream.
    4. Drain stdout and stderr in a loop using `reproc::drain` or manual `read` calls.
    5. Call `process.stop(...)` / `process.wait(...)` to retrieve exit status.
    6. Map `REPROC_ETIMEDOUT` to `timed_out = true`.
    7. Detect signal kill (on POSIX, exit status from reproc will be negative or a
       specific sentinel; document platform behaviour in a comment).
  - **Review Criteria:** Unit tests cover: successful execution + stdout capture, non-zero
    exit code, stdin data fed to child, timeout kill with partial output, missing
    executable path returns OS error.

- [ ] **Task 4: Wire Scanner + Runner into `clibridge/main.cpp`**
  - **Description:** Refactor `main.cpp`:
    1. Parse `--dir <path>` from `argv` (simple manual parse; no extra library).
       If `--dir` is absent, print usage to `stderr` and `exit(1)`.
    2. Call `scan_executables(dir)`. If empty, log warning and continue.
    3. For each `{tool_name, abs_path}` pair, register an MCP tool:
       - Name: `tool_name` (relative path string, e.g. `"util/convert.sh"`).
       - Description: `"Executes " + tool_name`.
       - One optional string parameter: `"stdin"` â€” data to pipe into the process.
    4. Each tool handler:
       a. Extracts optional `stdin` param (default `""`).
       b. Checks `std::filesystem::exists(abs_path)` â†’ if false, throw
          `mcp_exception(invalid_params, "Executable no longer exists: ...")`.
       c. Calls `run_process(abs_path, stdin_data)`.
       d. Builds response JSON:

            "stdout": "...",
            "stderr": "...",

## đź›ˇ Edge Case & Safety Checklist

- [ ] Recursive scan does not follow symlinks into infinite loops (`follow_directory_symlink`
      must remain false, which is the `std::filesystem` default).
- [ ] Tool names containing path separators are valid MCP tool name strings (no
      sanitization needed per spec, but verify with MCP client).
- [ ] Zombie processes: reproc handles `waitpid`; do not call it independently.
- [ ] SIGPIPE: on POSIX, ignore `SIGPIPE` in `main()` before starting server
      (`signal(SIGPIPE, SIG_IGN)`) to prevent crashes when child closes stdin pipe early.
- [ ] Concurrent calls to the same executable: each call spawns an independent process
      instance; no shared state.
- [ ] Large output: no artificial buffer cap; `std::string` accumulation. Document that
      callers should be mindful of very large outputs.
- [ ] Windows compatibility: `std::filesystem::perms::owner_exec` may not reflect true
      executability on Windows (all files may appear executable). Add a `#ifdef`-guarded
      fallback using `GetBinaryType` or simply check file extension (`.exe`, `.bat`,
      `.cmd`, `.ps1`). Mark as TODO if Windows is out of scope for this iteration.
